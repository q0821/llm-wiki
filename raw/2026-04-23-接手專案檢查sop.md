# WordPress 接案交接：如何檢查「前人的遺產」

> 最近接手一個 WordPress 專案，遇到一些奇怪的狀況記錄下來，順手整理一下

---

## 各種奇怪的狀況決定要來檢查一下

最近接手一個網站，一開始就覺得哪裡不對勁：

1. 明明是**形象官網**，竟然裝著 **WooCommerce + XStore 電商主題**
2. 外掛總數接近 **60 個**，而且有很多功能重複的 —— 光是「嵌入第三方追蹤碼」這類用途就放了 **4 個外掛**
3. 明明主機用的不錯了，但**後臺操作編輯還是卡成狗**
4. 編輯文章儲存後，內容會**自動還原**回改之前
5. 媒體庫上傳檔案，跑去了**前年的資料夾**而不是今年的

想說怎麼可能這樣...這太詭異了吧！所以決定來看看到底怎麼回事 —— 不然製作中一直卡住、還原被各種詭異的事情打斷，心情就阿雜。

於是做了一輪檔案系統檢查，找到三個歷史文物：

- 一個完整的 LINE Bot 範例外掛，`config.ini` 內有**真實的 Channel Access Token 與 Secret**
- 一個測試簡訊發送的 php 放在網站根目錄，裡面有**第三方簡訊平台的真實帳密**
- `wp-content/uploads/` 底下有個 `wph/environment.php`，把**原本的伺服器路徑** `/var/www/[別家網域]/` 寫死在 JSON 裡

看起來就像是之前的設計公司，複製了別的站台，拿來改成這個專案。
我把這種狀況叫做「**前人的遺產**」。這篇就是我整理出來的檢查 SOP，方便未來接手類似案子時能更快識別。

---

## 什麼是「前人的遺產」

接案工程師為了省時間，接到新案時直接複製舊專案當基底，改個主題、換個內容就交件。沒清乾淨的東西可能包括：

- 舊客戶的 API 帳密、Token、Secret
- 舊客戶的伺服器絕對路徑、網域名稱
- 測試用腳本（根目錄直接放 `.php` 測試檔是最常見的）
- 多餘外掛（舊案曾經裝來解某個需求，新案根本用不到）
- 資料庫裡的舊資料殘留

對新業主來說，這造成：

| 風險類型 | 具體表現 |
| --- | --- |
| **效能負債** | 一堆不用的外掛還在載入、跑排程、佔記憶體 |
| **相容性衝突** | 功能重複的外掛互相干擾（前面「編輯自動還原」很可能就是這類原因） |
| **資安風險** | 承接了別家的帳密，網站可能變成洩密管道 |
| **道德尷尬** | 撿到別家的憑證，要怎麼處理？ |

## 檢查 SOP

### Step 1：用 WP-CLI 驗證核心檔完整性

最快的方法是讓 WordPress 自己說話。WP-CLI 會向 wordpress.org 下載**該版本的官方 checksums**，跟實際檔案比對：

```bash
wp core verify-checksums
```

**一切正常時**會看到：

```
Success: WordPress installation verifies against checksums.
```

**有問題時**會列出三種狀況：

| 訊息 | 意義 | 處理 |
| --- | --- | --- |
| `File doesn't verify against checksum` | 核心檔被改過 | **高度警戒**，可能是 malware 植入 |
| `File should not exist` | 多餘檔，不屬於核心 | 前人的遺產常犯區（例如測試簡訊發送的 php、殘留範例外掛會被抓到） |
| `File is missing` | 核心檔遺失 | 極少見，通常表示有人手動刪過東西 |

這一個指令就同時覆蓋了「多餘檔盤點」與「核心檔被竄改」兩件事，比 find + grep 的手動白名單更可靠。

補充外掛完整性檢查（對官方 WordPress.org 目錄的外掛有效，商業外掛如 XStore、Elementor Pro 會 skip）：

```bash
wp plugin verify-checksums --all
```

**若沒裝 WP-CLI** 或沒辦法執行（例如只有 FTP 存取權），退回用 find 列出根目錄所有 `.php`，手動比對 WordPress 核心檔案列表：

```bash
find . -maxdepth 1 -type f -name "*.php"
```

WordPress 根目錄正常只會有這 17 個 PHP：`index.php`、`xmlrpc.php`，以及 `wp-activate.php`、`wp-blog-header.php`、`wp-comments-post.php`、`wp-config.php`、`wp-config-sample.php`、`wp-cron.php`、`wp-links-opml.php`、`wp-load.php`、`wp-login.php`、`wp-mail.php`、`wp-settings.php`、`wp-signup.php`、`wp-trackback.php`。多的就要逐個檢查。

### Step 2：檢查 `wp-content/uploads/` 內有無 PHP 檔（最重要）

`uploads/` 是 WordPress 主要上傳圖片的目錄。此處若有 `.php`，要不是外掛自動建的「防目錄列表」空檔，就是**可疑的檔案**。

```bash
find wp-content/uploads -type f \
  \( -name "*.php" -o -name "*.phtml" -o -name "*.phar" \
     -o -name "*.pht" -o -name "*.php5" -o -name "*.php7" \) 2>/dev/null
```

對每個結果，**打開看內容**：

| 內容樣態 | 判斷 |
| --- | --- |
| 檔案 size 只有幾 bytes，只有 `<?php` 一行或 `die();` | 外掛的防列表空檔，**正常** |
| 可讀的 JSON 設定、環境變數 | 可能前案設定殘留，**可疑**（本案的 `wph/environment.php` 屬此類） |
| 看不懂的 eval / 亂碼 / base64 字串 | **立即隔離，八成是後門** |

### Step 3：在自訂區域搜尋硬編碼帳密

排除 vendor 與知名外掛後，grep 可疑字串：

```bash
grep -rniE "(password|passwd|api_key|api-key|apikey|secret|access_token|auth_token|bearer)\s*[:=]\s*['\"][^'\"]{6,}['\"]" \
  . --include="*.php" --include="*.ini" --include="*.conf" --include="*.env" \
  --exclude-dir=wp-admin --exclude-dir=wp-includes \
  --exclude-dir=node_modules --exclude-dir=vendor \
  2>/dev/null | head -80
```

合法外掛中這些字串多出現在函數定義、變數名、i18n 字串 (`__('password')`)，要打開看 context。重點是找 `password = 'xxxxxxxx'` 這種**字面指派**。

### Step 4：翻出殘留設定檔內的「原始伺服器路徑」

這一步是前人的遺產的**決定性證據**。複製過來的 code 可能在 cache、設定、log 中寫死了原始部署路徑，例如 `/var/www/[原網域]/...` 或 `/home/[原帳號]/...`。

```bash
grep -rn "/var/www/\|/home/[a-z]" wp-content/ \
  --include="*.php" --include="*.json" --include="*.ini" --include="*.log" \
  2>/dev/null | head -20
```

找到一個**不是當前網域**的路徑，基本確定有前人的遺產。

### Step 5：資料庫也可能有殘留

需要資料庫存取權。最快的方法是從 options 表與 post content 搜起：

```sql
-- options 表中是否有別家網域殘留
SELECT option_name, LEFT(option_value, 100) FROM wp_options 
  WHERE option_value LIKE '%[可疑網域]%' LIMIT 20;

-- post content 中是否有別家絕對連結
SELECT ID, post_title FROM wp_posts 
  WHERE post_content LIKE '%[可疑網域]%' LIMIT 20;
```

另一個常被忽略的地方是 **serialized data**（WordPress 外掛大量用這個存設定），純 SQL 搜尋會找到但**不能用 SQL UPDATE 直接改**，要用 WP-CLI 的 `search-replace --all-tables` 才能正確處理：

```bash
wp search-replace '舊網域' '新網域' --all-tables --dry-run
```

確認結果合理再去掉 `--dry-run`。

### Step 6：盤點外掛數量與功能重疊

```bash
ls wp-content/plugins/ | wc -l
```

多沒有關係，如果都是獨立的功能那沒有關係，但是如果多還功能重複就值得整理一下。
用 WordPress 後臺逐個審視：

- 類似功能的外掛（追蹤碼嵌入、備份、SEO、快取）有沒有裝超過一個
- 主題相關（例如 XStore 會帶一整套自家外掛）是否真的需要
- 停用一個月都沒人抱怨的外掛，通常可以直接移除
- 有時候也會有惡意的程式偽裝成外掛，然後寫入 DB， 再把自己從外掛列表隱藏起來，這個真的很賊。

## 發現可疑檔案後的處理原則

按風險由低到高分類：

1. **已知外掛/主題會自動生成** → 留著，刪了也會再生
   - 例：`cleantalk_fw_files_for_blog_1/index.php`（CleanTalk 防目錄列表用）
   - 例：`wpvividbackups/.htaccess`（WPvivid 備份外掛自建）

2. **前人的遺產但無功能關聯** → 直接刪
   - 例：測試簡訊發送的 php、獨立的 LINE Bot 範例資料夾

3. **前人的遺產但不確定主題/外掛是否依賴** → 先備份再刪，觀察幾天
   - 例：`uploads/wph/environment.php`（看起來是主題的 cache）

4. **看起來像後門** → **立即隔離、告知業主、考慮主機層級清毒**
   - 判斷特徵：eval、assert、`$_GET`/`$_POST` 直接執行、base64_decode 執行字串、obfuscated code、檔案時間戳與周邊檔案差異極大

## 倫理篇：撿到別家的帳密怎麼辦

這次檢查中我撿到了別家客戶的 LINE Token、簡訊 API 帳密。處理原則：

- **不利用**：不測試它還能不能用、不記錄到長期筆記、不在公開場合展示內容
- **不傳播**：對話結束就刪除，不留在交付文件中
- **視關係決定是否通知**：
  - 若新業主**認識**前一家客戶 → 請新業主去通知
  - 若新業主**不認識**（例如我這次）→ 無從通知，結案
  - 不要自己跨級去找不認識的公司講這件事，既無法證明善意、也可能造成法律糾紛

真正該負責的是**當初做複製的工程師**。撿到的人沒有義務替對方擦屁股，但有義務不讓它擴散。

## 給前手工程師的自我檢查清單

如果你是那個複製舊專案的工程師，交付前請至少做一次：

- [ ] 網站根目錄有沒有其他非 WP 核心 PHP 檔？
- [ ] `wp-content/uploads/` 內有沒有 `.php`？
- [ ] 所有設定檔 (`*.ini`、`*.env`、`*.json`) 內的 Token / 帳密是否**清空或換成新業主的**？
- [ ] 資料庫 options 表是否已用 WP-CLI `search-replace` 過舊網域？
- [ ] 是否刪除了與新業主無關的多餘外掛？
- [ ] log 檔、cache 是否已清空？
- [ ] 主題的 cache 目錄（例如 `uploads/[主題名]/` 或 `uploads/wph/`）是否已清除？

這七點做到位，就不會把自己搞成「洩密中介」。
